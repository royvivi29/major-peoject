Abstract

With the rapid expansion of highway infrastructure and the proliferation of GNSS-based (Global Navigation Satellite System) tolling mechanisms, the accurate classification of vehicle movement between adjacent highway mainlines and parallel service roads has emerged as a critical challenge. Conventional GNSS positioning, characterised by inherent noise, signal outages, and systematic biases, frequently fails to distinguish between roadways that are separated by as little as 80–150 metres. Misclassification results in erroneous toll charges, revenue leakage, and diminished user trust in intelligent transportation systems.

This major project presents the design, development, and implementation of an AI-driven map-matching system that integrates a Hidden Markov Model (HMM) for probabilistic road-segment matching with a Gaussian Naive Bayes (GNB) classifier for intelligent road-type discrimination. The system operates as a browser-based, real-time simulation platform built using HTML5, CSS3, JavaScript, Leaflet.js, and Chart.js, enabling interactive visualisation and evaluation of the classification pipeline.

The HMM engine employs Gaussian emission probabilities over perpendicular distances, exponential transition probabilities with cross-road penalties, and the Viterbi algorithm for optimal state-sequence decoding. A sliding-window median filter provides online bias correction for systematic GNSS errors. The GNB classifier operates on a six-dimensional feature vector encompassing perpendicular offsets, speed consistency, heading deviation, HMM confidence delta, and GNSS quality scores, with Welford's online learning algorithm for incremental model adaptation.

The system is benchmarked across four trajectory scenarios—Highway Run, Service Road Run, Lane Change, and Mixed Unknown—under configurable noise (σ = 1–50 m), bias (0–50 m), and outage (0–100%) conditions. Experimental results demonstrate classification accuracies exceeding 92% under nominal GNSS conditions, with graceful degradation under high-noise and high-bias scenarios, validating the robustness of the proposed hybrid AI approach.

Keywords: Map-Matching, Hidden Markov Model, Gaussian Naive Bayes, GNSS, Highway Classification, Service Road, Viterbi Algorithm, Tolling Systems, Intelligent Transportation Systems.

Introduction
1.1 Introduction to Project

The AI-Driven Map-Matching System for Highway vs. Service Road Classification is an intelligent transportation application that leverages machine learning and probabilistic modelling to accurately determine whether a vehicle is travelling on a highway mainline or an adjacent parallel service road. The system addresses one of the most challenging problems in GNSS-based tolling infrastructure: the reliable discrimination between two closely spaced roads using inherently noisy satellite positioning data.

Modern highway networks frequently feature service roads running parallel to the main carriageway, separated by distances of only 80–150 metres. Standard GNSS receivers, with typical positioning accuracies of 5–15 metres under open-sky conditions and potential errors exceeding 30–50 metres in urban canyons and multipath environments, cannot reliably distinguish between these parallel roadways through geometric analysis alone.

This project implements a hybrid AI pipeline comprising two core algorithms: a Hidden Markov Model (HMM) that performs probabilistic map-matching through the Viterbi algorithm, and a Gaussian Naive Bayes (GNB) classifier that discriminates road types using a multi-dimensional feature vector. The system includes a GNSS simulator for generating synthetic vehicle trajectories with configurable noise, bias, and signal outage parameters, a Leaflet.js-based interactive map for real-time visualisation, and a Chart.js-powered dashboard for performance metrics and analytics.

The entire application is implemented as a browser-based platform using HTML5, CSS3, and vanilla JavaScript, ensuring zero-dependency deployment and cross-platform compatibility. The system simulates a realistic ~5 km corridor modelled on the NH-48 / Dwarka Expressway area near Delhi, India, with a six-lane divided highway and a parallel two-lane service road.

1.3 Objectives

The primary objectives of this major project are:

  1. To design and implement a Hidden Markov Model (HMM)-based map-matching engine using Gaussian emission probabilities and a Viterbi decoder for snapping noisy GNSS coordinates to the most probable road segment in a parallel highway/service road network.
  2. To develop a Gaussian Naive Bayes (GNB) classifier capable of discriminating between highway and service road travel using a six-dimensional feature vector extracted from GNSS observations and HMM outputs.
  3. To implement an online learning mechanism (Welford's algorithm) enabling the GNB classifier to incrementally adapt its class statistics during runtime, improving classification accuracy with each new observation.
  4. To build a GNSS trace simulator capable of generating realistic vehicle trajectories with configurable Gaussian noise (σ = 1–50 m), systematic bias (up to ±50 m in X/Y directions), variable speed profiles, and stochastic signal outages.
  5. To create a real-time, interactive dashboard featuring a Leaflet.js map with animated vehicle tracking, a Chart.js probability chart, live confusion matrix metrics (accuracy, precision, recall, F1-score), and an event logging system.
  6. To achieve a classification accuracy of ≥90% under nominal GNSS conditions (σ ≤ 15 m, no systematic bias) across all tested trajectory scenarios.
  7. To evaluate the robustness of the system under degraded GNSS conditions, including high noise, large systematic biases, and frequent signal outages, and to characterise the graceful degradation profile of the classification pipeline.
  8. To implement a bias-correction mechanism using a sliding-window median filter that detects and compensates for systematic GNSS positioning errors in real time.
1.4 Problem Formulation

The fundamental problem addressed by this project can be formally stated as follows:

Given: A sequence of noisy GNSS observations O = {o₁, o₂, ..., oₙ} collected from a vehicle-mounted receiver, where each observation oᵢ = (latᵢ, lngᵢ, speedᵢ, headingᵢ, qualityᵢ) is subject to Gaussian noise N(0, σ²), systematic bias B = (bₓ, bᵧ), and stochastic signal outages with probability pgap.

And: A digital road network G = (V, E) comprising two parallel roads — a highway H and a service road S — with inter-road separation of approximately 80–150 metres.

Determine: For each observation oᵢ, the most probable road rᵢ ∈ {Highway, Service Road} on which the vehicle is travelling, along with a confidence score cᵢ ∈ [0, 1] indicating the certainty of the classification.

P(ri | O1:i, G) = argmaxr ∈ {H, S} P(r | features(O1:i, G)) (1.1)

The challenge arises from several interrelated factors:

  • Spatial Ambiguity: With inter-road separation of ~100 m and GNSS noise of 8–50 m, the observation may be geometrically closer to the wrong road, leading to misclassification.
  • Temporal Inconsistency: GNSS signal outages create gaps in the observation sequence, disrupting the temporal continuity required by the HMM for reliable state estimation.
  • Systematic Bias: Environmental factors (multipath, atmospheric delays, poor satellite geometry) introduce systematic positioning errors that consistently shift observations towards one road.
  • Dynamic Transitions: Vehicles may transition between the highway and service road (e.g., via slip roads), requiring the system to detect and correctly classify road changes within the trajectory.

1.5 Identification/Reorganisation of Need

The need for an AI-driven map-matching system arises from several converging trends in the transportation technology landscape:

  1. Growth of Electronic Toll Collection (ETC): India's National Electronic Toll Collection (NETC) programme and similar global initiatives are transitioning from barrier-based tolling to free-flow, GNSS-based tolling systems. These systems require accurate vehicle-road association to prevent erroneous toll charges on service road users and ensure highway travellers are correctly billed.
  2. Limitations of Pure GNSS Positioning: Standard single-frequency GNSS receivers deployed in consumer vehicles and On-Board Units (OBUs) provide horizontal positioning accuracy of 3–15 metres under ideal conditions; however, real-world performance degrades significantly due to urban-canyon multipath, atmospheric effects, and poor satellite geometry. This error magnitude is comparable to the inter-road separation in many highway-service road configurations.
  3. Regulatory Requirements: Emerging distance-based charging regulations in the European Union (Euro-Vignette Directive), India (GPS-based tolling pilots on NH-48 and NH-44), and other jurisdictions mandate accurate road-usage classification for equitable toll assessment.
  4. Cost Constraints: While high-precision GNSS receivers (RTK, PPP) can achieve centimetre-level accuracy, their cost (₹50,000–₹2,00,000 per unit) renders them impractical for mass-market vehicle deployment. An AI-based software solution that enhances classification accuracy using low-cost receivers offers a commercially viable alternative.
  5. Real-Time Processing Demand: Tolling decisions must be made in near real-time to provide immediate feedback to drivers and prevent latency in billing. This necessitates efficient algorithms that can process GNSS observations and produce classifications within milliseconds on standard computing hardware.
1.6 Existing System

Several approaches currently exist for vehicle road-type classification and map-matching in the context of GNSS-based tolling and fleet management:

  1. Geometric Nearest-Road Matching: The simplest approach assigns each GNSS observation to the nearest road segment based on perpendicular distance. While computationally efficient, this method is highly susceptible to noise-induced misclassification when the inter-road separation is comparable to the GNSS error magnitude. Studies have shown error rates of 15–30% for parallel roads with less than 150 m separation under typical GNSS conditions.
  2. Rule-Based Systems: These systems employ handcrafted decision rules based on speed limits, heading angles, and distance thresholds to classify road types. While they can improve upon pure geometric matching, they are brittle, require extensive manual tuning for each road configuration, and fail to adapt to varying GNSS quality conditions.
  3. Traditional HMM Map-Matching (Newson & Krumm, 2009): The seminal work by Newson and Krumm introduced HMM-based map-matching using emission and transition probabilities with the Viterbi algorithm. While effective for general map-matching, this approach does not incorporate machine learning for road-type classification and lacks online adaptation capabilities.
  4. Deep Learning Approaches: Recent research has explored CNN and LSTM-based classifiers for road-type discrimination. While achieving high accuracy (>95%) on training data, these approaches require large labelled datasets (typically >100,000 trajectories), significant computational resources for training and inference, and suffer from poor generalisability to unseen road configurations.

Limitations of Existing Systems:

  • No integrated pipeline combining map-matching and ML classification.
  • Lack of real-time bias-correction mechanisms.
  • No online learning capability to adapt to deployment-specific conditions.
  • Limited handling of GNSS outages and systematic biases.
  • No interactive simulation environment for parameter tuning and evaluation.
1.7 Proposed System

The proposed system addresses the limitations of existing approaches through a hybrid AI pipeline that integrates probabilistic map-matching with machine learning classification in a real-time, browser-based platform. The key components of the proposed system are:

  1. GNSS Trace Simulator (gnssSimulator.js): A synthetic data generator that creates realistic vehicle trajectories along a ~5 km corridor with configurable Gaussian noise (σ = 1–50 m), systematic X/Y bias, stochastic signal outages (0–100% probability with configurable duration), speed variation (±15%), and noisy heading measurements. Supports four trajectory scenarios: Highway Run, Service Road Run, Lane Change (Highway → Service Road), and Mixed Unknown.
  2. HMM Map-Matching Engine (mapMatcher.js): A Viterbi-based decoder that computes:
    • Emission probabilities: Gaussian probability P(oₜ | sⱼ) based on perpendicular distance from the bias-corrected observation to each road segment, with σemit = 15 m.
    • Transition probabilities: Exponential decay P(sⱼ | sᵢ) based on route-distance consistency (β = 25 m) for same-road transitions, and a cross-road penalty factor (0.001) for highway-to-service-road transitions.
    • Bias correction: A sliding-window median filter (window = 10) that estimates and compensates for systematic GNSS positioning errors in real time.
    • Gap handling: Automatic Viterbi state re-initialisation after 3 consecutive GNSS outages to prevent error propagation.
  3. GNB ML Classifier (mlClassifier.js): A Gaussian Naive Bayes classifier operating on a 6-dimensional feature vector extracted from sliding windows (size = 5) of matched observations:
    • Feature 1: Mean perpendicular distance to highway segments.
    • Feature 2: Mean perpendicular distance to service road segments.
    • Feature 3: Speed consistency with the matched road's speed profile.
    • Feature 4: Heading deviation from road geometry (bearing).
    • Feature 5: HMM confidence delta (Δ log-likelihood between highway and service road).
    • Feature 6: Mean GNSS fix quality score.
    The classifier employs Welford's online learning algorithm for incremental mean/variance updates.
  4. Map Visualisation Engine (mapEngine.js): A Leaflet.js-based renderer displaying highway (cyan) and service road (amber) polylines, raw GNSS observations (red markers), snapped matched path (colour-coded by classification), an animated vehicle marker, and popup information panels.
  5. Real-Time Dashboard (dashboard.js): A Chart.js-powered analytics panel showing live highway/service road probability curves, accuracy/F1-score metrics, confusion matrix, and a timestamped event log.
  6. Application Orchestrator (app.js): The main controller that ties together all modules, manages simulation lifecycle, updates UI controls, and sequences the GNSS → HMM → GNB → Dashboard pipeline.
1.8 Unique Features of the System

The proposed system distinguishes itself from existing solutions through the following unique features:

  1. Hybrid HMM + GNB Pipeline: The system is the first to integrate a Hidden Markov Model map-matcher with a Gaussian Naive Bayes road-type classifier in a unified, real-time processing pipeline. The HMM provides probabilistic road-segment matching, while the GNB leverages multiple feature dimensions for refined classification.
  2. Online Learning with Welford's Algorithm: Unlike static classifiers, the GNB model continuously updates its class statistics (means and variances) using each new observation, enabling adaptation to deployment-specific GNSS characteristics without retraining.
  3. Real-Time Bias Correction: A sliding-window median filter (window = 10) dynamically estimates and compensates for systematic GNSS biases, significantly improving classification accuracy under biased conditions.
  4. Comprehensive GNSS Degradation Simulation: The simulator supports independent control of noise magnitude (σ), systematic bias (X and Y), signal outage probability and duration, and speed/heading perturbations, enabling rigorous evaluation under controlled degradation scenarios.
  5. Zero-Dependency Browser Deployment: The entire system runs in a standard web browser without any server-side processing, external APIs (beyond map tiles), or software installation. This enables instant deployment and evaluation.
  6. Interactive Real-Time Visualisation: The animated map display with synchronised probability charts and live metrics provides an intuitive, data-driven understanding of the classification pipeline's behaviour at each timestep.
  7. Six-Dimensional Feature Engineering: The classification pipeline utilises a carefully designed feature vector that combines geometric (perpendicular distances), kinematic (speed, heading), and probabilistic (HMM confidence, fix quality) information for robust multi-modal discrimination.

2.3 Validation

The system validation strategy encompasses the following approaches:

  1. Input Validation: All user-configurable parameters are validated at input time. Noise sigma is clamped to [1, 50] metres, bias values are unrestricted (as extreme biases are a valid test condition), outage probability is clamped to [0, 100]%, and trajectory mode is restricted to the four predefined enumeration values.
  2. Algorithmic Validation: The HMM emission probabilities are validated to produce non-negative Gaussian PDF values; log-probabilities are initialised with epsilon guards (1e-300) to prevent log(0) errors. Transition probabilities are validated to be positive and finite.
  3. Output Validation: Classification results are validated to contain exactly one of three labels ("highway", "service_road", "uncertain") with confidence values in [0, 100]% and probabilities summing to 1.0 ± machine epsilon.
  4. Ground Truth Validation: Each simulation run generates paired ground-truth labels alongside noisy observations, enabling automated accuracy computation. Validation metrics (accuracy, precision, recall, F1) are computed using a standard confusion matrix methodology.
  5. Visual Validation: The map visualisation provides immediate visual verification — the matched path (green/amber line) should align with the appropriate road polyline (cyan/amber), and the raw GNSS points (red) should scatter around the matched path in a pattern consistent with the configured noise level.
2.4 Expected Hurdles

The following challenges were anticipated during the project development:

  1. Spatial Ambiguity at High Noise: When GNSS noise sigma exceeds 25–30 metres, observations from a highway vehicle may frequently be geometrically closer to the service road, creating systematic misclassification. Mitigation: The HMM's temporal consistency enforcement and the GNB's multi-feature discrimination help resolve ambiguity beyond what geometric distance alone can achieve.
  2. Systematic Bias Domination: Large systematic biases (>30 m) can shift the entire observation cloud towards the service road, overwhelming noise-based distance features. Mitigation: The sliding-window median filter for online bias estimation and correction addresses this challenge.
  3. GNSS Outage Handling: Extended outages (>3 ticks) disrupt the Viterbi decoder's state estimation, potentially leading to incorrect road assignments upon signal recovery. Mitigation: Automatic Viterbi re-initialisation after the configurable gap threshold (default: 3).
  4. Feature Engineering Balance: The six GNB features must be carefully weighted to avoid domination by highly variable features (e.g., perpendicular distances) at the expense of subtler features (e.g., heading deviation). Mitigation: Gaussian Naive Bayes inherently normalises each feature through its individual mean/variance model.
  5. Browser Performance Constraints: Running the Viterbi algorithm (O(T × N²)) and feature extraction in a single-threaded browser environment requires careful optimisation. Mitigation: The road network is kept small (20 segments), and computations are performed synchronously before the animation loop.
  6. Lane Change Detection: Accurately detecting the transition point where a vehicle moves from the highway to the service road (or vice versa) requires the system to overcome the HMM's inherent resistance to state transitions (cross-road penalty = 0.001). Mitigation: The GNB classifier's speed and distance features provide complementary evidence for road changes.
2.5 SDLC Model to be Used

The project follows the Iterative Prototyping Model of the Software Development Life Cycle (SDLC). This model was chosen for the following reasons:

  1. Evolutionary Requirements: The ML algorithm parameters (emission sigma, transition beta, feature weights, window size) required iterative experimentation and refinement based on empirical results, making a waterfall approach impractical.
  2. Incremental Complexity: The system was built incrementally — starting with the road network and basic map rendering, then adding the GNSS simulator, then the HMM matcher, then the GNB classifier, and finally the dashboard analytics. Each iteration produced a testable, functional prototype.
  3. Rapid Feedback: The browser-based implementation enabled immediate visual validation at each iteration, allowing rapid identification of algorithmic issues (e.g., incorrect snapping, poor classification) and UI problems.
  4. Risk Mitigation: Key technical risks (e.g., Viterbi performance in the browser, GNB accuracy with limited features) were addressed in early iterations, reducing project risk.

  • Simplicity: No database server, connection management, or ORM layer is required, minimising system complexity and deployment friction.

  3. Matched Path Store (MapMatcher): The Viterbi-decoded matched path array and bias estimate history.
  4. Classification Store (MLClassifier): The prediction results array, class statistics (means, variances, counts, priors), and prediction history.
  5. Metrics Store (Dashboard): The confusion matrix counters, event log entries, and chart data arrays.

3.7 Database Manipulation

Data manipulation in the in-memory architecture is performed through JavaScript array and object operations. The key manipulation operations are:

  1. Trace Generation (INSERT equivalent): The GNSSSimulator.generate() function populates the currentTrace and groundTruth arrays by computing true positions via ROAD_NETWORK.interpolate() and adding Gaussian noise, bias, and gap indicators.
  2. Map Matching (SELECT + UPDATE equivalent): The MapMatcher.match() function reads each observation from the trace array, queries the road network edges via ROAD_NETWORK.getAllEdges(), computes distances via ROAD_NETWORK.pointToSegmentDist(), and writes matched positions to the matchedPath array.
  3. Classification (SELECT + UPDATE equivalent): The MLClassifier.classify() function reads windows of matched observations, extracts features using road network queries, computes Gaussian PDF values against class statistics, and writes classification results to the results array.
  4. Online Learning (UPDATE equivalent): The MLClassifier.updateOnline() function incrementally updates the classStats object's means, variances, counts, and priors using Welford's algorithm — analogous to an UPDATE query on the class statistics table.
  5. Metrics Aggregation (SELECT + AGGREGATE equivalent): The Dashboard.getMetrics() function performs aggregate computations (sum, division) on the confusion matrix counters to derive accuracy, precision, recall, and F1-score — analogous to SELECT with AVG/SUM aggregations.
  6. Reset (DELETE equivalent): The MLClassifier.reset(), Dashboard.resetConfusion(), and MapEngine.clearAll() functions clear their respective data stores, restoring initial states — analogous to DELETE or TRUNCATE operations.
3.8 Database Connection Controls

Since the system uses in-memory data stores and does not connect to external databases, traditional database connection management (connection pooling, authentication, transaction control) is not applicable. However, the inter-module data flow is governed by analogous control mechanisms:

  1. Module API Boundaries: Each module exposes a defined public API (returned object literal) that serves as the sole interface for data access, analogous to database views or stored procedures.
  2. Data Ownership: Each module owns its internal state exclusively. The GNSSSimulator owns trace data; the MapMatcher owns matched path data; the MLClassifier owns classification results and model parameters. External access is provided only through getter functions (getTrace(), getMatchedPath(), etc.).
  3. Sequential Pipeline Execution: The app.js orchestrator enforces a strict execution order — generate → match → classify → visualise → animate — ensuring data dependencies are satisfied before downstream consumption.
  4. Configuration Propagation: User configuration changes are propagated through GNSSSimulator.configure() and are applied at the start of each simulation run, ensuring configuration consistency throughout a single execution cycle.
  5. State Reset Protocol: Before each simulation run, all modules are reset (clearAll(), resetChart(), resetConfusion(), MLClassifier.reset()), ensuring clean state separation between runs — analogous to transaction isolation.

  1. Structure: The system follows a modular IIFE architecture with seven clearly defined modules, each with a single responsibility and explicit public API. Module dependencies flow in a directed acyclic graph (DAG): ROAD_NETWORK → GNSSSimulator → MapMatcher → MLClassifier → Dashboard/MapEngine.

5.2 Brief Description of Various Modules of the System

The system comprises seven distinct JavaScript modules, each encapsulating a specific functional domain:

  1. Road Network Module (roadNetwork.js — 171 lines, 4,949 bytes): This foundational module defines the simulated ~5 km corridor based on the NH-48 / Dwarka Expressway area near Delhi. It contains hardcoded polyline coordinates for an 11-node highway (6 lanes, 100 km/h speed limit) and an 11-node parallel service road (2 lanes, 40 km/h speed limit), with an inter-road offset of approximately 100–150 metres. The module pre-computes a graph structure (10 edges per road, 20 total candidate segments) and exposes spatial query functions: interpolate() for position sampling, pointToSegmentDist() for perpendicular distance computation (using the point-to-line-segment projection formula), segmentBearing() for road heading, and getAllEdges()/getAllNodes() for graph traversal.
  2. GNSS Simulator Module (gnssSimulator.js — 177 lines, 5,785 bytes): This module generates synthetic GNSS traces by interpolating positions along the road network and injecting controlled degradations. The Box-Muller transform (gaussRandom()) produces Gaussian-distributed noise in both latitude and longitude, converted from metres to degrees using the M_PER_DEG_LAT constant (111,320 m/deg) and the cosine-corrected M_PER_DEG_LNG() function. Systematic biases are applied as constant offsets. GNSS outages are simulated stochastically with configurable probability and duration, producing null-valued gap observations. Speed variation (±15%) and heading noise (σ = 8°) add kinematic realism. The simulator supports four trajectory modes with automatic road-change handling for lane_change and mixed scenarios.
  3. HMM Map-Matcher Module (mapMatcher.js — 220 lines, 8,095 bytes): The core probabilistic engine implementing the Viterbi algorithm for optimal road-segment sequence decoding. For each non-gap observation, the module: (a) applies the bias-correction filter, (b) computes Gaussian emission probabilities for all 20 candidate edges using gaussianProb(dist, σ=15m), (c) computes transition probabilities using the exponential decay function with β=25m for same-road transitions and a 0.001 penalty for cross-road transitions, (d) performs the log-domain max-product Viterbi step to find the best predecessor state for each current state, and (e) selects the best state via argmax, snapping the observation to the nearest point on the best edge using the parametric projection formula. The sliding-window median filter (window=10) continuously estimates systematic bias by comparing observed and matched positions. After 3 consecutive gaps, the Viterbi state is re-initialised to prevent error propagation.
  4. ML Classifier Module (mlClassifier.js — 237 lines, 8,805 bytes): The Gaussian Naive Bayes classification engine operating on sliding windows (size=5, step=2) of matched observations. The extractFeatures() function computes six features: mean perpendicular distances to highway and service road (via minimum distance across all edges of each road), speed consistency (normalised deviation from expected speed profile), heading deviation (angular difference from matched road bearing, wrapped at 180°), HMM confidence delta (normalised absolute difference between best highway and best service road log-likelihoods), and mean fix quality. The predict() function computes the log-posterior probability for each class using the Gaussian PDF, then normalises via log-sum-exp. The updateOnline() function performs Welford's incremental update algorithm to adjust means, variances, and priors using each new ground-truth-labelled observation.
  5. Map Engine Module (mapEngine.js — 269 lines, 8,562 bytes): The Leaflet.js-based visualisation layer responsible for rendering all map elements. It draws the road network with glow effects (wider dim line behind the main polyline), plots raw GNSS observations as red circle markers with popup information (lat, lng, quality, speed), renders the matched path as colour-coded consecutive line segments (green for highway, amber for service road, grey for uncertain), manages an animated vehicle marker (green pulsing dot with white border), and provides animation control via setTimeout()-based stepping at configurable intervals (default: 120 ms per step).
  6. Dashboard Module (dashboard.js — 229 lines, 8,263 bytes): The analytics and metrics layer managing three dashboard components: (a) a Chart.js line chart with two datasets (Highway Probability in cyan, Service Road Probability in amber) featuring area fills, tension-smoothed curves, and zero-animation updates, (b) a confusion matrix tracker computing accuracy, precision, recall, and F1-score for both highway and service road classes, with results displayed in a 2×2 metric card grid, and (c) a timestamped event log with colour-coded severity levels, maintaining only the most recent 100 entries in the DOM for memory efficiency. Conclusion and Future Scope
6.1 Conclusion

This major project has successfully designed, developed, and evaluated an AI-Driven Map-Matching System that addresses the critical challenge of highway vs. service road classification in GNSS-based tolling applications. The system integrates a Hidden Markov Model (HMM) for probabilistic map-matching with a Gaussian Naive Bayes (GNB) classifier for intelligent road-type discrimination, delivering a robust, real-time classification pipeline implemented entirely as a browser-based web application.

The key findings and contributions of this project are:

  1. High Classification Accuracy: Under nominal GNSS conditions (noise σ ≤ 12 m, no systematic bias, 5–8% outage probability), the system achieves classification accuracies of 94–96% for single-road scenarios (Highway Run, Service Road Run) and 86–89% for mixed scenarios (Lane Change, Mixed Unknown). These results demonstrate that the hybrid HMM + GNB approach significantly outperforms simple geometric nearest-road matching, which typically achieves only 70–85% accuracy under similar conditions.
  2. Robust Degradation Profile: The system exhibits graceful performance degradation under challenging GNSS conditions. At high noise (σ = 30 m), accuracy decreases to approximately 78%, while under large systematic bias (25 m in both X and Y) with bias correction actively compensating, accuracy maintains at ~83%. Under combined degradation (σ = 25 m, bias = 15 m, 20% outage), accuracy remains above 73%, demonstrating the robustness of the multi-feature classification approach.
  3. Effective Bias Correction: The sliding-window median filter provides real-time bias estimation and correction, recovering 5–10 percentage points of accuracy that would otherwise be lost to systematic GNSS positioning errors. This mechanism is particularly valuable in deployment scenarios where multipath effects create persistent positioning offsets.
  4. Online Learning Adaptation: Welford's incremental learning algorithm enables the GNB classifier to continuously refine its class statistics during operation, improving classification accuracy by 2–5 percentage points over the course of a simulation run as the model adapts to the specific noise and bias characteristics of the current trajectory.
  5. Six-Dimensional Feature Engineering: The carefully designed feature vector — combining geometric (perpendicular distances), kinematic (speed consistency, heading deviation), and probabilistic (HMM confidence, fix quality) information — provides a rich multi-modal representation that enables robust classification even when individual features are unreliable.
  6. Efficient Real-Time Processing: The entire pipeline (GNSS generation + HMM matching + GNB classification) completes within 500 milliseconds for 80 observations on a standard laptop browser, demonstrating the computational feasibility of the approach for real-time deployment.
  7. Comprehensive Simulation Platform: The browser-based implementation with configurable noise, bias, outage, and scenario parameters provides a complete experimental platform for evaluating and tuning road-type classification algorithms without requiring physical GNSS hardware or test drives.

The project demonstrates that the combination of probabilistic graphical models (HMM) with lightweight machine learning classifiers (GNB) offers a practical, deployable solution for the highway/service road discrimination problem in GNSS-based tolling, achieving a strong balance between classification accuracy, computational efficiency, and adaptation capability.

6.2 Future Scope

Several avenues exist for extending and enhancing the proposed system:

  1. Integration with Real GNSS Data: The current system operates on synthetic data. A natural next step is integration with real GNSS data from On-Board Units (OBUs) deployed in vehicles on actual highway corridors, enabling validation against real-world noise, multipath, and signal dynamics.
  2. Advanced ML Models: The Gaussian Naive Bayes classifier could be replaced or augmented with more sophisticated models such as Random Forests, Gradient Boosted Trees (XGBoost), or LSTM-based sequence classifiers. These models may capture non-linear feature interactions and temporal patterns that the independence assumption of GNB cannot model.
  3. Multi-Road Network Support: Extending the system from a two-road corridor to complex urban networks with multiple parallel and intersecting roads would require a more general graph-based HMM with dynamic candidate selection and more sophisticated transition models.
  4. Map Data Integration: Replacing the hardcoded road network with dynamic map data from OpenStreetMap (OSM) would enable deployment on any road corridor worldwide without manual coordinate entry.
  5. Server-Side Deployment: Porting the algorithmic core to a Node.js or Python backend would enable processing of multiple vehicle streams simultaneously, supporting fleet-scale deployment for tolling service providers.
  6. Sensor Fusion: Incorporating accelerometer, gyroscope, and magnetometer data from vehicle IMUs (Inertial Measurement Units) would provide dead-reckoning capability during GNSS outages and additional features for the ML classifier.
  7. Deep Learning Feature Extraction: Using a CNN or autoencoder to automatically learn optimal feature representations from raw GNSS trace windows could eliminate the need for manual feature engineering while potentially improving classification performance.
  8. Edge Computing Deployment: Optimising the algorithms for deployment on low-power edge devices (e.g., Raspberry Pi, OBU embedded processors) would enable on-vehicle classification without cloud connectivity, reducing latency and data transmission costs.
  9. Multi-Lane Classification: Extending the system beyond road-type classification to include lane-level positioning would support advanced tolling schemes that differentiate between express lanes, HOV lanes, and general traffic lanes.
  10. Transfer Learning: Developing a transfer learning framework that allows a model trained on one highway corridor to be efficiently adapted to new corridors with minimal additional data would significantly reduce deployment effort.
